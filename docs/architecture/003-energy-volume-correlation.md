# ADR-003: Energy-Volume Correlation (에너지-거래량 상관관계)

| Field | Value |
|-------|-------|
| **Status** | Accepted |
| **Date** | 2026-04-01 |
| **Decision Maker** | user + game-designer |
| **Relates To** | price-engine.md (Rule 4-2), 프로토타입 검증 결과 |

## Context

주식 거래량은 가격 변동의 강도와 상관관계가 있다. 급등/급락 시 거래량이 폭증하고,
횡보 시 거래량이 감소하는 것이 현실 시장의 패턴이다. 게임에서 이 패턴이 없으면
차트가 비현실적으로 보이고, 거래량 분석이 무의미해진다.

프로토타입(`prototypes/price-engine/`)에서 여러 거래량 모델을 테스트한 결과,
"tick energy" 기반 모델이 가장 자연스러운 차트를 생성했다.

## Decision

**Tick Energy 모델**을 채택한다. 가격 변동을 일으키는 힘(force)의 절대값 합을
"에너지"로 정의하고, 에너지가 높을수록 거래량이 증가한다.

### 공식

```
tick_energy = |pattern_delta| + |event_delta|
energy_mult = 1.0 + clamp(tick_energy / ENERGY_THRESHOLD, 0.0, ENERGY_MAX_BOOST)
final_volume = base_volume × state_mult × energy_mult × limit_dampen × tod_mult
```

### 핵심 설계: 힘의 절대값 합 (상쇄 전)

`pattern_delta`가 +5%이고 `event_delta`가 -3%이면:
- 최종 가격 변동: +2% (상쇄 후)
- tick_energy: |5%| + |3%| = 8% (상쇄 전)

이것이 핵심이다. 상반되는 힘이 작용하면 가격은 적게 움직이지만 거래량은 크다.
현실 시장에서 "가격 변동은 적은데 거래량이 폭증"하는 패턴(세력 매집, 방향 전환 전조)을
자연스럽게 재현한다. 이로써 플레이어가 거래량으로 숨겨진 시장 역학을 읽을 수 있다.

### 추가 거래량 팩터

| 팩터 | 역할 | 범위 |
|------|------|------|
| `state_mult` | 마르코프 상태별 기본 거래량 배율 (SURGE=2.5, DRIFT=0.5 등) | 0.5~2.5 |
| `energy_mult` | 에너지 기반 증폭 | 1.0~4.0 |
| `limit_dampen` | 가격제한(±30%)에 근접 시 거래량 감소 (호가 고갈) | 0.1~1.0 |
| `tod_mult` | 시간대별 배율 (개장/마감 근처 높음) | 1.0~2.5 |

## Alternatives Considered

### A. 가격 변동 절대값 기반 (상쇄 후)

```
volume_factor = |final_price_change|
```

- 장점: 단순
- 단점: 상반되는 힘이 작용할 때 가격은 0 변동인데 거래량도 0 → 비현실적.
  "거래량으로 숨겨진 힘을 읽는" 게임플레이 불가능

### B. 이벤트 기반 거래량 (이벤트 있을 때만 증가)

- 장점: 구현 단순
- 단점: 이벤트 없는 날에 거래량이 일정 → 비현실적.
  마르코프 상태 전환에 따른 자연스러운 거래량 변화 표현 불가

### C. 랜덤 거래량 (가격과 무관)

- 장점: 구현 최소
- 단점: 거래량 분석이 게임플레이에 아무 가치가 없음.
  스킬 트리의 거래량 분석 해금이 무의미해짐

## Consequences

### 긍정적

- 차트가 현실 시장과 유사한 가격-거래량 패턴을 보임 (프로토타입 검증 완료)
- 거래량 분석이 실제 게임플레이 가치를 가짐 (숨겨진 힘 감지)
- 스킬 트리의 거래량 관련 스킬 해금이 의미있는 파워 증가로 작용
- 가격제한 근접 시 거래량 감소로 "상한가 도달 후 거래 고갈" 현상 재현

### 부정적

- 거래량 공식이 단순 랜덤보다 복잡 (5개 팩터의 곱)
- 튜닝 노브가 많아 밸런스 조정 시 상호작용 고려 필요
- ENERGY_THRESHOLD, ENERGY_MAX_BOOST 값은 프로토타입에서 경험적으로 결정 → 추후 재튜닝 가능성
